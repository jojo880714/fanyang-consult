# 語言學校比較頁面 — 內容結構提案 v2

> 給工程師對接用：固定 HTML 模板 + 資料 placeholder（無 AI 自由生成）

**頁面用途**：留學顧問展示給學生看，比較 2–5 間語校
**比較對象單位**：校區（同一學校的不同城市可以同時比）
**觀眾**：準留學生（看頁面決定）+ 顧問（用頁面說服）

---

## 總覽：現有 sections 的處置

| # | 現有 section | 處置 | 說明 |
|---|---|---|---|

## 新增 sections（5 個）

| # | 新 section | 為什麼新增 |
|---|---|---|
| A | TLDR 一句話差異 | 學生開頁 5 秒內知道每校的「身份」，不用滑到底 |
| B | 教學品質與政策 | english_only_policy / class_size / accreditation 是顧問說「為什麼貴」的核心 |
| C | 適合哪種你 | suitable_for[] 結構化呈現，學生對號入座 |
| D | 國籍與班級組成 | nationality_count 是「會不會班上都台灣人」的決定性回答，須獨立 |
| E | 下一步 / 預約 | 顧問用頁面說服後需要落地動作 |

### Section 1 — Hero

**目的**：告訴學生「現在在比較哪幾間」，並可現場加減學校。對顧問來說，是諮詢開場的視覺定錨。

**視覺**：

```
│                                                │
│   COMPARING                                    │

│   3 間語言學校・5 個校區                       │
│                                                │
│   為你比較最適合的選項                          │
│   (H1, 32px, weight 700)                       │
│                                                │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│   │ EP 倫敦   │ │ Kaplan   │ │ EF 布萊  │      │
│   │ ✕        │ │ 多倫多 ✕  │ │ 頓 ✕    │      │
│   └──────────┘ └──────────┘ └──────────┘      │
│                                                │
│   [+ 加入比較]   [清空]                        │
│                                                │
└────────────────────────────────────────────────┘
```

**資料來源**：
- `schools.name` + `campuses.city`（現有）
- 計數：`COUNT(DISTINCT school_id)` 與 `COUNT(campus_id)`（衍生）

**條件邏輯**：
- 1 校：標題改「先認識這間學校」，隱藏加減 chips
- 2–5 校：正常顯示，每 chip 可移除
- > 5 校：顯示「最多比 5 間」warning，超過部分 disable

**互動**：chip 點 ✕ 移除該校；「+ 加入比較」開 modal/dropdown 加學校（純前端，列表來自全 schools）

**手機調整**：< 640px chip 改成 2 欄堆疊

**優先級**：**必有**

## 給工程師的整體建議

### 1. 資料層需要新增的欄位（共 5 個）

| 欄位 | 屬於 | 用在哪 | 性質 |
|---|---|---|---|
| `one_liner` | schools | Section 2 | 文字 < 40 字，手寫 |
| `english_only_policy_label` | schools | Section 3 | 文字 < 12 字 |
| `nationality_breakdown[]` | schools | Section 10 | `[{flag,name,pct}]`，定期手動更新 |
| `includes` | housing | Section 8 | 文字（含早餐/半膳…）|
| `commute_to_school` | housing | Section 8 | 文字（步行 X 分） |

### 2. 全域常數需建立

- **學員人物 master list**（Section 4 用）：8–12 個人物 tag
- **匯率 table**（Section 7 用）：`{ '£': 40, '€': 35, 'USD': 32, 'CA$': 23, 'AU$': 21 }`
- **放洋 contact info**（Section 11 用）：LINE / phone / booking_url / working_hours

### 3. 狀態管理

- 目前比較的校區 set（影響全頁）
- 學費試算的 program 選擇（每校一個）+ 週數
- 校區 tab 的 active（Section 9）

### 4. URL state

建議至少把以下兩個寫進 URL params，方便「複製連結傳給家人」這個關鍵動作：
- 比較中的校區（`?campuses=ep-canary,kaplan-toronto,ef-brighton`）
- 學費試算選擇（`&weeks=12&program-ep-canary=classic-morning`）

### 5. RWD 三斷點

| 寬度 | 行為 |
|---|---|
| < 640px | mobile：所有多欄改 1 欄；tab 改 horizontal scroll；按鈕 vertical stack |
| 640–1024px | tablet：表格保持但 sticky 第一欄；卡片 2 欄 |
| ≥ 1024px | desktop：完整版面 |

### 6. 信任 / 真實性原則

任何「估算」「參考」性質的數字（國籍比例、機票費、生活費、班級組成）必須有可見的 disclaimer。EP 校區內部政策（退費比例、轉校手續）不寫具體數字，改以「依合約為準 / 請洽顧問」處理。

避免：捏造的公司資訊、學員見證、班級組成統計、退費百分比 — 學員看到「待業主提供」「N=120 樣本」這種偽精確資訊會直接懷疑可信度。

## 從消費者視角看的優先排序（執行順序）

如果工程師需要分階段做：

**Phase 1（決策關鍵，先做）**：1, 2, 4, 5, 7, 10
- Hero + TLDR + 適合誰 + 總覽表 + 學費試算 + 國籍 = 學生 / 顧問當下做決定需要的全部

**Phase 2（深入細節，第二批）**：3, 6, 8, 9
- 教學品質、課程方案、住宿矩陣、校區城市 = 第二輪諮詢深入用

**Phase 3（落地動作，最後）**：11
- CTA 預約 = 顧問說服完的下一步

依此順序開發，每階段都是可獨立上線的版本。
